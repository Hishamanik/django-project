Analysis and design of the web application: The first step in part 4 was designing a schema. Django provides models to interact with databases instead of directly working with them, but I have added a schema file called schema.sql which contains the raw sql statements. Below is an pseudo-ER diagram of the tables. Briefly, the User table is Django's built in user (used primarly for authentication). The store user contains a reference to the user, and the cart and order tables contain a total price and reference to the store user. Computers are broken down into two tables. The Computer table contains the name price, and wether it is a default computer or not. Default computes appear in the store (and non-default ones are assigned to users). Computers can be related to a users cart or order, and also have a specification. In this part, I first changed up api.py to use Django's QuerySet API to interact with the models. After replacing the old code interacting with the JSON file, I added functionality for creating a computer which is assigned to a user. This would allow me to populate a user's cart and order. For authentication, I used Django's build in system, which uses the User model. This required me to develop login/signup screens. Once a user logs in, they can access all screens in part 4. If a user is not logged in, requests to any path other than /login or /create are redirected to the login page.
